# MCP Server Module

Part of mist_automation — see root `CLAUDE.md` for global architecture and conventions, `backend/CLAUDE.md` for backend patterns. See also `app/modules/llm/CLAUDE.md` for the LLM/agent layer that uses this server.

## Backend (`app/modules/mcp_server/`)

- FastMCP server exposing app data as MCP tools (backups, workflows, executions, webhook events, reports, system stats)
- HTTP endpoint (`/mcp`) served via **streamable HTTP** (`mcp.http_app(path="/")` — FastMCP v3 default transport). Works for in-app chat and for **external MCP clients** (Claude Desktop, VS Code, Cursor).
- Gated by `MCPAuthMiddleware` (`auth_middleware.py`). Accepts two Bearer token types:
  - **JWT session tokens** — issued via `/api/v1/auth/login`, looked up against `UserSession`. Used by the in-app chat.
  - **Personal Access Tokens** (`mist_pat_…`) — long-lived, SHA-256 hashed, user-scoped, revocable. See `app/models/personal_access_token.py` and `app/core/pat.py`. PAT auth updates `last_used_at` via a fire-and-forget background task. Managed by users in the Profile → Tokens tab (`/api/v1/users/me/tokens`). Max count per user: `SystemConfig.max_pats_per_user` (default 10).
- Token-type detection is prefix-based (`mist_pat_` → PAT path, everything else → JWT path). Returns 401 for unauthenticated requests.
- In-process memory transport (`InProcessMCPClient`) bypasses HTTP entirely — unaffected by auth middleware
- **External MCP client setup**: users create a PAT in Profile → Tokens, then configure their client. Claude Desktop example:
  ```json
  {"mcpServers":{"mist":{"url":"https://<host>/mcp","headers":{"Authorization":"Bearer mist_pat_..."}}}}
  ```
  Tools run as the token's owning user with that user's full role set — same ownership and security checks as the in-app path.
- `mcp_user_id_var` and `mcp_thread_id_var` ContextVars for user and thread context in tool handlers
- **Elicitation bridge** (`helpers.py`): `elicit_confirmation()` sends simple text confirmations via WebSocket; `elicit_restore_confirmation()` sends rich payloads with `elicitation_type` and `data` fields for structured UI (e.g., diff viewer in restore confirmation card)
- **Backup restore action**: `backup(action="restore")` auto-computes diff between target and current version, sends rich elicitation with diff data, executes restore on approval
- **`_version_detail` smart pruning** (`tools/backup.py`): when called WITHOUT `fields`, the tool passes `inline_keys = {top-level segment of each entry in changed_fields}` to `prune_config`. This auto-expands whatever actually changed in that version (e.g. `port_config.ge-0/0/8` → full `port_config` content) so the LLM sees the interesting bits on the first call. Only applies on data-event versions (full_backup/updated/created/incremental) where `changed_fields` is populated.
- **`prune_config` extended** (`helpers.py`): supports `inline_keys: set[str] | None` to force full recursion into named keys, and `small_dict_threshold: int = 3` for auto-expanding small nested dicts/lists. Backwards compatible — callers passing no kwargs get essentially the same output as before, except nested dicts with ≤3 keys now render inline instead of as `"{...} (1 keys)"` summaries.
- **Workflow tool security** (`tools/workflow.py`): `_create` and `_update` mirror REST API security: role/ownership checks via `mcp_user_id_var`, `_encrypt_nodes()` for OAuth/auth secrets before persisting, `validate_no_circular_subflow_references()` on create. Uses same `_OAUTH_NODE_TYPES` set as the REST router.
- **Validation + errors policy**: MCP tools with action/type dispatch or cross-field dependencies validate coherence at runtime and raise `fastmcp.exceptions.ToolError` for invalid combinations, unresolved placeholders, and unsupported values. Avoid returning `{"error": ...}` payloads for input validation failures.
- **Shared validation helpers** (`tools/utils.py`): use `is_placeholder()`, `is_uuid()`, and `endpoint_has_placeholder()` instead of re-implementing placeholder/UUID checks in each tool.
- **Search tool guardrails** (`tools/search.py`): strict type-aware parameter validation (`search_type`, `object_type`, `site_id`, `status`, `event_type`, `hours`, sorting). For site-name lookup, use `search_type='backup_objects'` with `object_type='sites'` or `object_type='info'`. The primary parameter is `search_type` (renamed from `type` to avoid shadowing the Python builtin and to pair cleanly with `object_type`).
- **Duplicate-name disambiguation** (`tools/search.py`): `_search_backup_objects` adds a `duplicate_names` dict and `disambiguation_hint` to the response ONLY when 2+ results share the same `name`. Each entry lists `{id, status, summary, date}` for the colliding objects. Additive — unique-query responses are unchanged. Pure helper: `_collect_duplicate_names(results)`.
- **Backup search returns org_id/site_id** (`tools/search.py`): `_search_backup_objects` emits `org_id` and `site_id` on every result so the LLM can round-trip them to `digital_twin` / `get_configuration_object_schema` without guessing. Prevents a failure mode where the LLM copies the object `id` from a search hit and reuses it as `org_id`, which compiles a valid-looking write that fails at runtime with a misleading "site not found in backup" error. Sourced from `BackupObject.org_id` / `BackupObject.site_id` via the `$group` projection — both may be `None` for legacy records.
- **Digital Twin default org_id** (`tools/digital_twin.py`): `_resolve_twin_org_id(explicit, default_org_id=None)` falls back to `SystemConfig.mist_org_id` when the caller omits `org_id`. The async tool entrypoint `await`s `SystemConfig.get_config()` once per call and passes the default through `_validate_twin_inputs` — keeps the validator sync (tests unchanged). LLM guidance is in the `org_id` Field description: "NEVER copy an id from a search result as org_id — use the default or the `org_id` key the search emits." Resolution order: explicit caller value > `default_org_id` > `ToolError`.
- **Configuration schema tool** (`tools/configuration_schema.py`): `get_configuration_object_schema(object_type, org_id, site_id?, object_id?, verbose?)` derives payload schema from Mist OpenAPI request/response contracts (OAS-first) and returns endpoint metadata plus a sanitized example payload. Call this BEFORE `digital_twin(action='simulate')` to build valid payloads from required fields and example payload output. The legacy `sample_size` parameter has been removed — it was never used with the OAS-derived path.
- **Field discovery tool** (`tools/configuration_schema.py`): `find_configuration_field(query, scope?, limit?)` is the FIRST step when the user asks to change a setting you can't immediately map to an `object_type`. It walks every supported `Object_type` schema in the cached Mist OAS spec, flattens each to leaves, and returns keyword-scored matches with `{object_type, path, description, type, enum, score}`. Backed by a lazy module-level reverse index (`_FIELD_INDEX_CACHE`) keyed by the OAS spec identity — rebuilds when the OAS cache turns over. Path convention: `parent.child` for objects, `parent[].child` for array items, `parent.*.child` for dict-of-object (`additionalProperties`). Full discovery chain: `find_configuration_field` → `get_configuration_object_schema` → `validate_configuration_payload` → `digital_twin(simulate)`.
- **Payload dry-validator** (`tools/configuration_schema.py`): `validate_configuration_payload(object_type, org_id, payload, site_id?, action_type?)` walks a draft payload against the same OAS schema and returns a structured `{valid, errors, missing_required, warnings}` report. Use between `get_configuration_object_schema` and `digital_twin(simulate)` to catch type/required-field mistakes without consuming a twin session slot. Raises `ToolError` only for invalid tool arguments (bad UUIDs, placeholders in org_id/site_id) — validation failures are returned as data.
- **Digital Twin tool guardrails** (`tools/digital_twin.py`): simulate supports BOTH contracts: (1) single-change fields (`action_type`, `object_type`, `org_id`, `site_id`, `payload`, `object_id`) and (2) multi-change `changes=[{action_type, object_type, site_id?, payload?, object_id?}, ...]` under one `org_id`. `changes` is mutually exclusive with single-change fields. Each change compiles into one validated Mist write operation. Session actions (`approve`, `reject`, `status`, `history`) reject simulate-only fields (including `changes`) to prevent ambiguous calls. Additional protections: payload placeholder detection, per-change field validation, `session_id` ObjectId format + placeholder validation, and response guidance fields (`execution_safe`, `next_action`, `compiled_writes`) to reduce LLM orchestration errors.
- **Digital Twin response contract** (`tools/digital_twin.py`): simulate and status responses ALWAYS include `execution_safe` (bool) and `next_action` (`'approve'` or `'fix_and_resimulate'`), even when no prediction report was generated — in that case `execution_safe=False`, `next_action='fix_and_resimulate'`, and a `warning` field is added. Simulate returns `requested_changes` + `compiled_writes` for multi-change sessions; single-change aliases `requested_change` + `compiled_write` are preserved for backward compatibility. Runtime observability fields are also emitted for admins/LLMs: `executed_checks` (per-check status+summary), `issues` (non-pass checks with details/remediation), `check_diagnostics` (full per-check payload including `description`, `pre_existing`, affected scope), and `decision_log` (ordered one-line trace per check). LLMs may branch on these fields without a None check.
- **Cross-user session access** (`tools/digital_twin.py`): when a user requests approve/status on a session they do not own, the tool logs a `twin_session_access_denied` warning server-side and raises `ToolError('Session ... not found')` — deliberately indistinguishable from a nonexistent session to prevent id-probing. Access control is via `mcp_user_id_var` matching `session.user_id`.
- **Singleton object types** (`tools/digital_twin.py`, `tools/configuration_schema.py`): `site_info` (PUT `/api/v1/sites/{site_id}`) and `site_setting` (PUT `/api/v1/sites/{site_id}/setting`) are singletons — no `object_id`, only `action_type='update'` is supported, `site_id` is the identifier. Routed via the `_SINGLETON_OAS_PATHS` dict in `digital_twin.py`, which both the twin path builder and `configuration_schema._find_oas_paths` consult as a shortcut before the generic OAS walker (for `site_info` the 4-segment path is invisible to the walker, so the override is the only way). **Semantic distinction (critical):** use `site_info` for template bindings (`networktemplate_id`, `rftemplate_id`, `gatewaytemplate_id`, `aptemplate_id`, `alarmtemplate_id`, `sitetemplate_id`, `secpolicy_id`, `sitegroup_ids`) and site identity (`name`, `timezone`, `latlng`, `address`); use `site_setting` for site-level runtime defaults (`auto_upgrade`, `wids`, `rogue`, `wifi`, `mist_nac`, `switch_mgmt`, `remote_syslog`, etc.). The two OAS schemas don't share field names, so `find_configuration_field` naturally routes queries to the right singleton — the distinction is ALSO documented on the tool `object_type` Field descriptions (`digital_twin`, `get_configuration_object_schema`, `validate_configuration_payload`) for cases where the LLM guesses without searching first.
- **Compound-word tokenizer** (`tools/configuration_schema.py::_tokenize_query`): splits long compound tokens (≥8 chars) ending in a known domain root (`template`, `profile`, `policy`, `setting`, `config`, `group`, `webhook`, `topology`, `rule`, `tag`) into their constituent parts while keeping the original token so exact-name matches still score. Example: `networktemplate` → `["networktemplate", "network", "template"]`. This lets the existing multi-term AND filter (`hit_count >= 2` suppression around lines 1390) distinguish `networktemplate_id` from the 500+ noise hits for plain `network`. Longest-suffix-first matching so plural roots (`templates`) win over singular (`template`).
- **Memory tools** (`tools/memory.py`): `memory_store(key, value, category)`, `memory_recall(query, category)`, `memory_forget(key)`. Per-user scoped via `mcp_user_id_var`. Store upserts by `(user_id, key)`. Recall uses MongoDB text index (keyword match, NOT semantic search). Cap enforced from `SystemConfig.memory_max_entries_per_user`. Invalid categories/keys and missing user context raise `ToolError`. Internal helpers `_store_memory`, `_recall_memory`, `_forget_memory` are exported for direct testing. Docstrings explicitly forbid storing secrets (plain text, full-text indexed).
- **`activate_skill` tool** (`tools/skills.py`): loads the full body of a named enabled `Skill` document from the filesystem. Returns content wrapped in `<skill_content name="...">` tags with `<skill_resources>` listing bundled files. Invalid skill names, missing files, and parse failures raise `ToolError`.
- **FastMCP annotation policy**: every tool declares `annotations={...}` on its `@mcp.tool(...)` decorator. Read-only tools (search, get_details, get_configuration_object_schema, validate_configuration_payload, memory_recall, activate_skill, search_impact_sessions, get_impact_session_detail) set `readOnlyHint=True, destructiveHint=False`. Tools that mutate state (workflow create/update, backup trigger/restore, digital_twin approve, memory_store/forget) set `destructiveHint=True`. `openWorldHint=True` for tools that hit the Mist cloud; `False` for purely local-DB tools. Annotations are hints for clients; actual security is enforced by `MCPAuthMiddleware` (HTTP JWT) plus per-tool role/ownership checks mirroring the REST API.
- **Action/param matrix docstring format**: dispatcher tools (backup, workflow, digital_twin) document each action as a block with `Required:`/`Optional:` sublists plus a short description. Keeps per-action coherence visible at a glance and gives the LLM a single source of truth instead of prose. New dispatcher tools should follow the same format.
