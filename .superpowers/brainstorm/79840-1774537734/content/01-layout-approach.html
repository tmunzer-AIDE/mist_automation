Impact Analysis Detail — Layout Approach

How should the chat-like AI narrative and the data sections relate to each other?

A

Chat-First with Expandable Details

The page IS the chat. The AI narrates the entire monitoring process as messages ("Starting baseline capture...", "Validation complete — 2 issues found"). Data sections (SLE table, validation checks) appear as expandable cards inline in the chat when relevant. When LLM is off, the timeline entries render as system messages instead.

Starting impact analysis for switch-lobby-01. Capturing baseline metrics...
🤖 AI Agent • 2:32 PM
Baseline captured. SLE metrics recorded (6 metrics). Waiting for config to be applied...
🤖 AI Agent • 2:35 PM
Config applied. Running validation checks...
⚠️ Validation Results • 2:40 PM
2 warnings found. Port flapping detected on ge-0/0/3. DHCP pool utilization at 92%.
▶ Click to expand full validation details
🤖 AI Agent • 2:40 PM
I detected issues during validation. Let me do a deeper analysis using MCP tools to understand the root cause...
🤖 AI Agent • 2:42 PM
Analysis complete. The port flapping on ge-0/0/3 correlates with the new VLAN config. The port's access VLAN was changed from 10 to 20, but the connected AP still expects VLAN 10. Recommend reverting the port VLAN or updating the AP WLAN config.
B

Split View: Chat Panel + Data Panel

Two-column layout. Left side is the AI chat/narrative (similar to the global chat). Right side has the structured data: progress bar, validation results, SLE metrics. The chat references the data, and clicking items in the chat scrolls/highlights the relevant data section.

C

Unified Timeline with AI Narration

Single-column feed (like a GitHub PR activity). Each event is a card: status changes, validation results, SLE snapshots, AI commentary. The AI messages weave naturally between system events. Compact, chronological, no separate sections. Data is always inline — no hidden panels to open.
